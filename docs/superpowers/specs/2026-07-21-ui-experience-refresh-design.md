# Evidence Graph 全局 UI 体验优化设计

## 1. 背景与目标

Evidence Graph 已完成作品集、托管代码、持久化研究结果和报告发布。当前页面功能完整，但视觉层仍存在系统性问题：大量 10–12px 文本降低可读性，公共页面使用 72–112px 的大段留白，首屏主任务不够直接，左侧色线被重复用于引用、选中态和状态提示，移动导航依赖压缩字号容纳全部入口，工作台控件和元数据层级过碎。

本里程碑不增加业务功能，也不改变研究、发布、权限或 Provider 契约。目标是在稳定功能基线上，统一全站的字体、密度、页面层级、响应式行为、交互反馈和状态页面，使公共页面体现 Ailian 的产品工程能力，使应用页面成为安静、清晰、可重复操作的研究工具。

## 2. 已确认方向

用户在三套方向中选择 **Neutral Product Studio**，并确认采用均衡密度。中文页面必须完整使用中文内容；英文页面继续保留一一对应的英文翻译。

### 2.1 备选方向与取舍

1. Editorial Research Desk：品牌辨识度强，但继续依赖衬线和编辑式构图，不符合用户选择。
2. Neutral Product Studio：公共页面清晰克制，工具页面密度稳定，最适合统一作品集与研究工作台，作为最终方向。
3. Evidence Console：技术感强，但全站深色和等宽字体不利于长文、公开报告与中文阅读。

### 2.2 视觉命题

中文优先的中性产品工作室：以矿物白和纸灰为基础，用近黑文本、森林绿主操作、朱红反驳/错误和低饱和黄色限定状态建立可信、清晰的证据产品界面。公共页面有明确品牌与真实产品画面，工具页面以扫描效率和重复操作为先。

### 2.3 内容计划

1. 首页：Ailian 品牌、具体工程定位、Evidence Graph 真实图谱和主要入口。
2. 作品与案例：项目结果、方法和当前证据，不使用营销式功能卡片。
3. 工程笔记：主题、状态、日期和验证结论形成可扫描列表。
4. 认证与项目：直接呈现登录、项目状态和下一步任务。
5. 研究工作台：主张、图谱/报告、来源和运行日志维持专业三栏结构。
6. 公开报告：围绕阅读、核查、分享和打印组织。

### 2.4 交互命题

1. 导航、按钮、筛选和面板切换使用 160–220ms 的颜色、边框、位移或透明度反馈；不使用 `transition: all`。
2. 图谱继续通过节点、边和来源同步表达真实产品能力，不增加装饰性动画。
3. 页面进入和状态切换保持克制；`prefers-reduced-motion` 下关闭非必要运动。

## 3. 设计系统

### 3.1 颜色

| 语义 | 建议值 | 用途 |
| --- | --- | --- |
| 页面背景 | `#f6f7f4` | 公共页和应用基础背景 |
| 主表面 | `#ffffff` | 表单、工具面板和重点内容 |
| 次级表面 | `#eef1ee` | 选中态、状态页和分区背景 |
| 主文本 | `#18201c` | 标题和正文 |
| 次级文本 | `#66706a` | 说明、时间和元数据 |
| 分隔线 | `#d8ded9` | 区块、列表和控件边界 |
| 主操作 | `#286458` | 主要按钮、验证状态和当前选项 |
| 主操作深色 | `#1f4f46` | hover/active |
| 反驳/错误 | `#b74c40` | 冲突证据、错误和危险动作 |
| 限定/警告 | `#a9792a` | 限定条件和需注意状态 |
| 图谱画布 | `#111713` | 图谱专注模式，避免整页深色 |

颜色不能单独承载状态，必须同时提供文本、图标或形状。不得新增渐变、装饰色块、光斑或一套页面一个色系。

### 3.2 字体与字号

- 移除当前 `Inter` 和公共页面的衬线依赖。
- 中文优先使用系统无衬线：`"PingFang SC"`, `"Microsoft YaHei"`, `ui-sans-serif`, `system-ui`。
- 英文使用同一系统无衬线，保持中英文布局一致。
- 数量、成本、日期和表格数字启用 `font-variant-numeric: tabular-nums`。
- 字号不随 viewport 连续缩放，letter spacing 固定为 `0`。

| 层级 | 桌面 | 移动 | 约束 |
| --- | ---: | ---: | --- |
| 品牌 H1 | 56px | 42px | 只用于首页的 `Ailian` |
| 公共页 H1 | 42–48px | 32–36px | 中文最多 2–3 行 |
| 区块标题 | 30–36px | 26–30px | 不在紧凑面板使用 |
| 页面/工具标题 | 22–28px | 20–24px | 工作台和项目页 |
| 面板标题 | 14–16px | 14–16px | 紧凑、稳定 |
| 正文 | 14–16px | 14–16px | 公开报告正文优先 16px |
| 辅助文本 | 12–13px | 12–13px | 不低于 12px |
| 图谱节点 | 10–11px | 10–11px | 仅画布缩放内容；详情面板提供可读全文 |

### 3.3 间距与尺寸

- 基础间距使用 4/8px 系列。
- 公共页面 section 垂直间距以 48–64px 为主，移动端 36–48px。
- 工具页区块使用 8–24px；取消 72–112px 的常规页面空白。
- 控件高度：主要按钮 42–44px，输入框 42–44px，紧凑工具按钮 36–40px。
- 触控目标不小于 40×40px。
- 固定格式工具通过 grid track、min/max 和 aspect-ratio 保持稳定，不因 hover、标签或加载状态改变尺寸。

### 3.4 容器、边框和圆角

- 页面 section 不包装为浮动卡片。
- 卡片仅用于重复项目、模态框和确实需要边界的工具。
- 页面主要层级依靠全宽背景、分栏、留白和 1px 分隔线。
- 圆角上限 6px；常规控件 3–4px。
- 不使用胶囊标签，除非元素本身是状态筛选或短状态值。
- 移除引用、状态、选中项和分区中的左侧装饰线；改用背景、内描边、顶部规则、编号和图标。
- 不新增通用阴影。需要层级时使用极轻、带绿色调的扩散阴影。

### 3.5 控件与反馈

- 熟悉工具操作使用 lucide 图标；图标按钮必须提供 `aria-label` 和 `title`。
- 模式使用 segmented control，二进制设置使用 checkbox/toggle，选项集使用 select/menu。
- 主操作使用森林绿实心按钮；次操作使用边框或文本按钮；危险操作使用朱红文字或边框。
- focus 使用 `:focus-visible`，不可用无替代的 `outline: none`。
- 异步错误靠近触发区域，使用 `role="alert"` 或 `aria-live`；加载文案使用省略号 `…`。

## 4. 逐页设计

### 4.1 全局导航与页脚

桌面导航保留品牌、首页、作品、笔记、Evidence Graph 和语言入口，并显示当前页状态。移动端不再把所有入口压进一行，改为品牌、语言和菜单按钮；菜单使用语义按钮、可见焦点、Esc 关闭和稳定触控目标。页脚减少空白，保留联系、GitHub、语言和证据优先声明。

### 4.2 首页 `/{locale}`

- H1 保持字面品牌 `Ailian`，不使用价值主张替代品牌。
- 中文副标题明确产品前端、全栈和 Agent 工作流能力。
- Evidence Graph 的实际交互图谱继续作为首屏主视觉，整体改为浅色产品框架，图谱本身可保留深色专注画布。
- 首屏包含查看 Evidence Graph 和联系入口，并在常见移动/桌面高度露出下一段作品内容。
- 降低 hero 最小高度，提升节点和说明文字，不使用 10–11px eyebrow 堆叠。

### 4.3 作品列表 `/{locale}/work`

- 项目使用结构化行，不做等高卡片网格。
- 信息顺序为项目名、具体结果/摘要、技术、状态和入口。
- 整行主入口进入案例；仓库链接作为次级图标命令。
- 缩减项目间空白，状态和标签保持至少 12px。

### 4.4 项目详情 `/{locale}/work/[slug]`

- 首屏集中项目名、摘要、状态、仓库/产品入口和关键证据。
- 问题、方案、验证使用不套卡片的分栏或全宽信息带。
- 移除重复 eyebrow 和模板化标签集合。
- Evidence Graph 案例优先展示实际图谱和公开报告入口；其他项目保持真实已验证内容，不编造数据。

### 4.5 工程笔记 `/{locale}/notes`

- 形成可扫描列表：状态、标题、摘要、日期和序号基线对齐。
- 标题使用 18–22px，日期和状态不低于 12px。
- 减少大段空白，保持文章行之间的清晰分隔。

### 4.6 证据预览 `/{locale}/evidence`

- 页面说明、操作入口和图谱形成一个连续工作区。
- 外层使用浅色产品框架，图谱保持深色画布以确保节点/边对比。
- 选中来源、主张和证据关系保持同步，提示与工具不遮挡画布。

### 4.7 登录 `/{locale}/auth/login`

- 不做营销 hero；使用紧凑、无装饰的登录面板。
- 明确 GitHub 登录、登录后可做的工作和未配置/失败状态。
- 保持一个主要动作，移除无用卡片层级和 72px 外边距。

### 4.8 项目列表 `/{locale}/app`

- 桌面使用结构化项目行；标题和问题是主入口，状态和更新时间便于比较。
- 移动端自然折行为两层，不出现横向滚动。
- 归档/删除不与主入口竞争；危险操作仍要求确认。
- 空状态包含清晰的新建研究动作，不用大面积空卡片。

### 4.9 新建研究 `/{locale}/app/research/new`

- 桌面使用主问题与可选来源双栏；移动端恢复单栏。
- 标题、问题和语言在主栏，最多 5 个来源链接在次栏。
- 输入、说明和错误使用 14/12px 层级，错误贴近字段。
- 提交、pending 和稳定错误码不改变布局尺寸。
- 补齐非认证字段的 autocomplete 策略和必要的长 URL 断行。

### 4.10 研究工作台 `/{locale}/app/research/[id]`

#### 运行摘要

- 状态、研究问题和指标形成单一层级。
- 指标使用等宽数字和响应式网格；移动端自然换行，不靠缩小字号。

#### 主张列表

- 过滤器使用稳定 segmented control。
- 标签改用中文句式大小写，不使用重复大写 eyebrow。
- 选中态使用背景与内描边，不使用左侧色线。
- 状态通过图标、文字和颜色共同表达。

#### 图谱与报告

- 图谱/报告切换、工具栏和图例提升字号与触控尺寸。
- 图谱固定画布尺寸，继续验证 canvas 非空和节点交互。
- 报告版本、状态、发布/撤销、打开和复制按任务顺序排列。
- 报告章节、citation 与 Claim/Source 联动保持不变。

#### 来源与日志

- 引用使用浅色内容面和编号，不使用左侧红线。
- 元数据使用定义列表，长 URL 可断行。
- 运行日志入口和完成信息保持固定高度，展开不遮挡主工作区。

### 4.11 工作台状态页

queued、running、failed、loading、error 和 not-found 使用与完成态一致的页面骨架和稳定尺寸。每种状态包含当前发生了什么、数据是否保留、下一步动作；只有投递失败提供重投。状态变化不得造成页面跳动。

### 4.12 公开报告 `/r/[slug]`

- 压缩首屏空白，标题、研究问题、版本、发布时间和结论数量形成明确阅读入口。
- 正文约 16px，行宽控制在约 65 个中英文字符的可读范围。
- 引文通过编号、底色和分隔线表达，移除所有左侧红线。
- 分享/打印工具在桌面位于页头右侧，移动端位于标题后方；图标提供 tooltip 和可访问名称。
- 404 使用相同品牌头部与返回入口；打印继续隐藏工具并保留正文与来源。

## 5. 组件与代码边界

- 不修改领域模型、Repository、数据库 RPC、RLS、Inngest 或 Provider。
- 复用现有 React 组件和 CSS Modules；只有在多个页面确实共享交互时才提取组件。
- `SiteHeader` 可增加移动菜单和 active navigation，但不引入全局状态库。
- `globals.css` 负责 token、全局排版和公共页面系统；项目、工作台、登录和报告继续由对应 CSS Module 管理。
- 更新 `DESIGN.md` 以反映已选择的中性无衬线方向。
- 不引入 UI 框架、动画库或新字体依赖，优先使用系统字体和现有 lucide-react。

## 6. 可访问性、国际化与内容边界

- 中文页面所有可见文本使用中文；英文路由保留完整对应键，不在同一页面混排说明文案。
- 品牌名、技术名、代码和稳定标识可保留英文，并按需要使用 `translate="no"`。
- 页面包含跳转到主内容的 skip link和层级正确的标题。
- 表单控件拥有 label、name、正确 type/inputMode、autocomplete 和内联错误。
- 移动菜单、tabs、segmented controls 和图谱键盘交互必须可用。
- 所有 hover 状态同时提供 focus-visible 状态。
- 长中文、长英文单词、长 URL、空数组和错误消息不得溢出父容器。

## 7. 测试策略

### 7.1 RED-GREEN

- 移动导航、active navigation、表单可访问性和新增交互先补失败测试。
- 纯视觉变更先收紧 Playwright 布局/字号/溢出断言或建立失败截图基线，再修改 CSS。
- 每个页面组先跑聚焦 unit/E2E，保持功能测试不回归。

### 7.2 路由与视口

视觉验证覆盖 390×844、1024×768、1440×1000：

- `/zh`、`/en`
- `/zh/work`、至少一个项目详情
- `/zh/notes`
- `/zh/evidence`
- `/zh/auth/login`
- 项目列表和新建研究组件/本地认证页面
- `/zh/app/research/demo` 的图谱与报告模式
- `/r/traceable-citations-review-zh`
- loading、failed、not-found/404 代表状态

### 7.3 视觉断言

- 无横向溢出、文字裁切、控件重叠、异常空白和动态内容引发的布局跳动。
- 页面主要可见文本除图谱节点外不低于 12px。
- 首页首屏露出下一段作品内容。
- 选中态、引用和分区不使用左侧装饰线。
- 图谱 canvas 像素非空，节点/边结构和键盘交互不回归。
- 移动菜单、按钮和图标命令的触控目标稳定。

### 7.4 完整门禁

运行 Provider 边界、数据库、Schema lint、lint、typecheck、全部 unit、生产 build 和全部 E2E。测试环境显式禁用真实 Supabase/Inngest/Sentry/OpenAI/Tavily 连接，不调用真实或付费 Provider。

## 8. 交付方式

本次是跨页面设计系统和交互里程碑，使用 `feat/ui-experience-refresh` 独立 worktree 和唯一 Draft PR。实现按页面组使用中文 Conventional Commits；完整门禁和三档视觉复查通过后创建 PR，执行独立 Claude 审核、必要修复、重新审核、GitHub CI 和 merge commit 闭环。

## 9. 非目标

- 不增加新研究功能、计费、团队、Provider 或部署资源。
- 不改变研究结果、报告发布、权限和数据契约。
- 不引入暗色模式、主题切换、复杂动画、第三方 UI 框架或字体服务。
- 不为视觉效果生成不代表真实产品的装饰图像。
- 不在本里程碑处理 Vercel 账号恢复和生产发布门禁。

## 10. 完成标准

1. 所有列出页面和状态符合中性产品工作室方向，中文优先且英文路由完整。
2. 字号、间距、颜色、控件和状态层级由统一 token 驱动。
3. 既有产品流程、图谱联动、报告发布和匿名报告行为不回归。
4. 三档截图、键盘访问、长文本、溢出和 canvas 检查通过。
5. 完整 `test:managed` 通过，未调用真实 Provider。
6. 唯一 Draft PR 的独立 Claude 审核和 GitHub CI 对当前 head 均通过后，以 merge commit 合并。
