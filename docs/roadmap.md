# Evidence Graph 路线图：本地 MVP 收口到 Production Beta

更新时间：2026-07-24

## 1. 当前结论

Evidence Graph 已经具备 Auth、项目、研究工作流、证据工作台、报告发布和真实 Provider 等主要技术能力，但产品还不能标记为 MVP 完成。

当前缺口不是再搭一套生产基础设施，而是把已有能力整理成用户可以在本地连续验收的完整产品：

- 本地 Next.js、托管开发 Supabase、轻量 Inngest 和真实 Provider 尚未形成经过完整验收的运行链路。
- 核心研究闭环还没有经过用户本地完整验收和 P0/P1 问题收敛。
- 原产品计划中的 `/app/settings`、账号与数据删除尚未完成。
- 10 个固定问题的 Evidence Eval 尚未建立。
- 3 个真实公开案例和作品集回填尚未完成。
- 尚未形成经过用户确认的本地 Release Candidate。

Production 已有一份历史基线，但从现在起冻结。日常开发使用本地应用和当前托管开发数据库，不启动本地 Supabase Docker；只有 C6 本地 Release Candidate 验收通过，并再次取得用户明确授权后，才能执行 R1。

## 2. 完成度口径

后续不再用“代码已合并”代替“产品已完成”。每个能力分别记录三个状态：

| 维度 | 含义 | 完成条件 |
| --- | --- | --- |
| 代码完成度 | 功能是否已经实现并通过自动化门禁 | 相关测试、lint、typecheck、build 和视觉门禁通过 |
| 本地验收度 | 用户能否在本地看到并完成真实操作 | Agent 启动完整本地服务，用户按清单明确验收通过 |
| 产品完成度 | MVP 范围是否形成可发布的连续体验 | 所有必需里程碑完成，C6 Release Candidate 通过 |

Production 部署状态单独记录，只说明某个提交是否上线，不参与判断日常开发是否完成。

## 3. 执行规则

1. 同一时间只允许一个有限里程碑处于“进行中”。
2. 每个里程碑开始前单独编写设计说明和实施计划，不提前写跨越多个里程碑的巨型实现计划。
3. 每个里程碑必须写清进入条件、用户可见结果、自动化门禁、本地验收清单、明确不做和终止条件。
4. 实现与自动化门禁通过后，Agent 必须启动本地服务并提供 URL 与验收清单。
5. 完整实现和自动化门禁通过后，一个里程碑只创建一个 Draft PR。用户明确本地验收通过前，不合并 PR，不把状态写成“已完成”，也不开始下一里程碑。
6. 用户发现的问题先进入当前里程碑缺陷清单；只修复阻塞本里程碑目标的 P0/P1 问题，其他建议进入后续候选，不静默扩展范围。
7. C6 以前禁止更新 `release`、执行 Production 数据库迁移、修改 Production 环境变量、同步 Production Inngest 或重新部署。
8. 真实 Provider 调用不进入常规测试；每次真实运行都必须通过专用确认门禁和正数成本上限，且不得输出或提交密钥、请求正文、来源全文或付费 Provider 原始响应。

## 4. 里程碑总览

| 顺序 | 里程碑 | 用户可见结果 | 当前状态 |
| ---: | --- | --- | --- |
| C0 | 规划重置与真实进度基线 | 能清楚看到已完成、未完成、当前工作和停止条件 | 已完成 |
| C1 | 本地真实研究运行环境 | 从本地登录到完成一条低范围中文真实研究 | 进行中 |
| C2 | 核心研究闭环与缺陷收敛 | 创建、运行、核验、审核、发布全过程稳定可用 | 未开始 |
| C3 | Settings 与账号/数据生命周期 | 用户能管理语言、查看账号并安全删除项目数据或账号 | 未开始 |
| C4 | Evidence Eval 质量门禁 | 10 个固定问题有可重复的质量评测结果 | 未开始 |
| C5 | 真实案例与作品集回填 | 本地可查看 3 个真实案例和完整 Evidence Graph 案例页 | 未开始 |
| C6 | 本地 Release Candidate | 一个经过完整自动化和用户验收的可发布候选版本 | 未开始 |
| R1 | Production Beta 发布 | 明确批准的 Release Candidate 更新到 Production | 冻结，等待 C6 |

## 5. C0：规划重置与真实进度基线

### 目标

把“技术能力已经存在”和“产品已经完成”分开，建立后续可感知、可停止、可验收的推进方式。

### 交付

- 本路线图。
- 精简后的 `PROJECT_STATUS.md`，首屏直接显示当前里程碑和下一次验收。
- 更新后的 `docs/development-plan.md` 与 `AGENT.md`，固化单里程碑和用户本地验收门禁。
- 一份生产先于本地完备度的流程复盘。

### 自动化门禁

- `git diff --check`
- 文档占位符扫描无未解释的 `TODO`、`TBD` 或 `FIXME`
- 路线图、状态页和 Agent 规则对当前阶段、Production 冻结和验收顺序表述一致

### 明确不做

- 不修改产品代码。
- 不配置本地或 Production 密钥。
- 不调用真实 Provider。
- 不更新 `release` 或触发部署。

### 终止条件

文档验证通过并提交到 `main` 后，C0 结束。下一次开发从 C1 的独立设计与实施计划开始。

## 6. C1：本地真实研究运行环境

### 目标

让用户通过固定本地入口连接托管开发数据库，在一个 C1 和一个 Draft PR 中完成登录与项目列表、fixture 研究闭环和低范围真实研究，全程不启动本地 Supabase Docker，也不借助 Production 部署调试。

### 进入条件

- C0 已完成。
- 当前托管 Supabase 明确作为 C1-C6 的开发数据库，仓库 migration 链继续作为 Schema 事实来源。
- 开始任何付费调用前再次确认本次成本上限。

### 用户可见结果

1. 通过现有 GitHub OAuth 登录并看到托管开发数据库中的项目列表。
2. 使用单 worker Inngest 从 UI 完成一条不产生费用的 fixture 研究。
3. 在明确成本和资源上限下完成一条低范围中文真实研究。
4. 看到 queued、running、ready 或明确失败状态。
5. 打开真实来源、主张、证据关系、运行日志和带引文草稿报告。
6. C1 全部实现和自动化门禁完成后只创建一个 Draft PR，再由用户一次性完成上述本地验收；验收前不合并。

### 自动化门禁

- 为托管开发环境隔离、OAuth loopback、Provider 成本与资源门禁和轻量启动脚本补充测试。
- 主工作区的运行配置不依赖已完成模块的旧 worktree；本地密钥文件保持 Git 忽略并限制为当前用户可读写。
- 现有 Provider fixtures、数据库、Auth、Inngest 和研究闭环测试保持通过。
- 完整 `npm run test:managed` 通过，默认路径不调用真实 Provider。
- 专用真实验收只允许一条最多 4 个来源、40,000 字符、20 个 embedding 批次和 `0.15 USD` 的研究，并记录脱敏的次数、状态和估算费用。

### 本地验收清单

- Agent 提供本地 URL，不要求用户手工拼接服务地址。
- 用户能登录、创建研究并观察状态推进。
- 用户能从 Claim 跳到对应 Evidence 和 Source quote。
- 用户能打开草稿报告并点击引文返回来源。
- 页面没有阻塞流程的错误、空白或无法恢复状态。

### 明确不做

- 不修复所有产品体验问题；发现的问题形成 C2 的有限清单。
- 不实现 Settings、账号删除、Evidence Eval 或真实案例内容。
- 不启动本地 Supabase Docker，不启用托管 anonymous 登录。
- 不修改 Production，不更新 `release`。
- 不把真实密钥写入仓库或终端输出。

### 终止条件

自动化门禁通过，用户在本地完成一条中文真实研究并明确验收通过。随后停止，不在同一周期开始 C2。

## 7. C2：核心研究闭环与 P0/P1 缺陷收敛

### 目标

围绕“创建研究 -> 运行 -> 检查证据 -> 审核 Claim -> 发布/撤销报告”这一条核心链路，关闭 C1 发现的阻塞和严重体验问题。

### 进入条件

- C1 已通过用户本地验收。
- C1 的观察记录已整理为有限问题清单，并按 P0、P1、P2 分类。

### 用户可见结果

- 核心链路没有 P0/P1 问题。
- 加载、失败、重试、空状态和长文本都有清楚且可恢复的表现。
- 中文与英文界面的功能和关键信息一致。
- 桌面与移动端均可完成核心任务。

### 自动化门禁

- 每个成立的问题先复现 RED，再完成最小 GREEN。
- 聚焦测试覆盖 agreed P0/P1 清单。
- `npm run test:managed` 和 390x844、1024x768、1440x1000 视觉门禁通过。
- 本地 fixture 完整闭环和一条受控真实研究回归通过。

### 本地验收清单

- 用户按完整核心链路重新操作一次。
- 用户确认 C1 记录的 P0/P1 问题均已关闭。
- 用户确认没有新的核心流程阻塞问题。

### 明确不做

- 不处理不影响主链路的 P2 视觉偏好和功能建议。
- 不新增导出、团队、支付、定时研究或通用聊天。
- 不修改 Production，不更新 `release`。

### 终止条件

有限问题清单中的 P0/P1 全部关闭，自动化门禁和用户本地复验通过。随后停止。

## 8. C3：Settings 与账号/数据生命周期

### 目标

完成原产品计划中的 `/app/settings`、语言偏好、项目数据删除和账号删除，使用户能控制自己的身份与研究数据。

### 进入条件

- C2 已通过用户本地验收。
- 删除语义、级联范围和重新认证要求在本里程碑设计说明中冻结。

### 用户可见结果

- `/app/settings` 显示当前账号信息和语言选择。
- 用户可以删除单个项目及其来源正文、向量、Claim、Evidence、Run、报告和相关审计可识别内容。
- 用户可以通过高风险确认流程删除整个账号和其拥有的数据。
- 删除后的私有页面、公开 slug 和旧会话行为符合明确预期。

### 自动化门禁

- 数据库约束、RLS、级联删除和跨用户隔离测试通过。
- Server Action、错误映射、重新认证/确认和双语 UI 测试通过。
- 删除相关 E2E、`npm run test:managed` 和三档视觉门禁通过。

### 本地验收清单

- 用户能修改语言并在再次访问时看到一致结果。
- 用户能删除测试项目，且旧公开报告不能继续访问。
- 用户能完成测试账号删除，并确认无法使用旧会话继续访问。

### 明确不做

- 不实现账单、团队、角色管理或完整数据导出。
- 不删除任何 Production 用户或数据。
- 不更新 `release`。

### 终止条件

本地测试账号的数据生命周期验收通过，且无孤立的公开内容或跨用户可见数据。随后停止。

## 9. C4：Evidence Eval 与证据质量门禁

### 目标

把“看起来能研究”变成可测量的质量结论，建立产品计划要求的 10 个固定研究问题和人工核验样本。

### 进入条件

- C3 已通过用户本地验收。
- 10 个问题的领域分布、人工标注格式和真实评测预算在设计说明中冻结。

### 用户可见结果

- 评测覆盖技术选型、产品竞品和市场事实。
- 每次评测输出 Quote 精确率、无引用事实段落数、Evidence Relation 抽样准确率、来源域名数、运行完成率和单次成本。
- 失败样本可以定位到具体 Run、Claim、Evidence 和 Citation，而不是只有总分。

### 自动化门禁

- Fixture eval 可重复运行且不外呼。
- 真实 eval 使用独立确认口令和总成本上限，不进入 CI。
- MVP 门槛保持为：Quote 精确子串 100%、公开报告无引用事实段落 0、Evidence Relation 抽样准确率不低于 90%、可用场景来源域名不少于 4、真实运行完成率不低于 90%、单次运行不超过 1 美元。
- 原始 Provider 响应和来源全文不进入 Git。

### 本地验收清单

- 用户能查看一份结构化评测摘要和具体失败样本。
- 用户抽查至少一个技术、竞品和市场问题的来源与关系判断。
- 未达门槛的指标有明确阻塞结论，不用文案解释为“通过”。

### 明确不做

- 不为提高分数更换产品范围或引入新的模型供应商。
- 不构建通用评测平台或在线排行榜。
- 不修改 Production，不更新 `release`。

### 终止条件

10 题评测完整执行，所有 MVP 门槛达到，用户完成本地抽查并明确通过。否则当前里程碑继续修复，不进入 C5。

## 10. C5：真实案例与作品集回填

### 目标

完成原产品计划中的首批 3 个真实研究案例，让作品集展示真实产品结果，而不是 fixture 或开发状态说明。

### 进入条件

- C4 质量门禁通过。
- 三个案例继续使用产品计划已确定的主题；如公开事实发生变化，只更新问题措辞和来源时间范围，不扩展案例数量。

### 用户可见结果

- 本地可打开 3 份真实 Evidence Graph 报告。
- 每个案例包含可检查的图谱、来源、引文、800-1500 字案例文章、决策图以及失败与修正记录。
- 首页和 Evidence Graph Case Study 使用真实工作台媒体和真实状态。

### 自动化门禁

- 所有公开事实段落至少有一个可打开引文。
- 公开页面不暴露账号、私有运行日志、Provider 密钥、请求正文或成本明细。
- 内容链接、404/撤销、SEO、打印样式、三档视觉和 `npm run test:managed` 通过。
- 只提交经过人工整理的公开内容和脱敏媒体，不提交付费 Provider 原始响应或来源全文。

### 本地验收清单

- 用户逐一打开 3 个案例，检查标题、结论、主要来源、图谱和引文。
- 用户从首页进入 Evidence Graph Case Study，再进入真实公开报告。
- 用户确认作品集描述没有虚构用户、收入或未实现能力。

### 明确不做

- 不新增第 4 个案例。
- 不新增 CMS、文章编辑器、社交分享系统或报告翻译。
- 不写入 Production 数据，不更新 `release`。

### 终止条件

3 个案例和作品集回填在本地通过用户验收。随后停止。

## 11. C6：本地 Release Candidate

### 目标

冻结一个可以发布的本地候选提交，证明 MVP 不依赖 Production 调试也能完整运行。

### 进入条件

- C1-C5 均已通过用户本地验收。
- 没有未处理的 P0/P1，P2 已形成明确延期清单。

### 用户可见结果

- 新环境按 README 可以启动完整本地产品。
- Auth、研究、证据审核、报告、Settings、删除、3 个案例和双语公共页面连续可用。
- 失败态、恢复路径、安全边界和成本提示与实际行为一致。

### 自动化门禁

- `npm run test:managed` 全部通过，常规路径不外呼。
- 关键 E2E 在干净本地数据库执行通过。
- 390x844、1024x768、1440x1000 全路由视觉矩阵通过。
- 专用真实研究和 Evidence Eval 仍满足确认门禁、成本上限和质量门槛。
- 生成 Release Candidate 验收记录，固定候选 commit SHA 和已知 P2 清单。

### 本地验收清单

- Agent 启动干净的完整本地环境并提供唯一入口 URL。
- 用户按发布验收清单完成一次完整 walkthrough。
- 用户明确判断该候选版本是否可以进入 Production Beta。

### 明确不做

- 不在 RC 阶段加入新功能。
- 不以“顺手修复”为由扩大到 P2 或路线图外需求。
- 未取得用户明确发布授权前，不更新 `release` 或任何 Production 配置。

### 终止条件

所有门禁通过，用户明确接受固定候选提交。若用户不接受，只记录具体问题并留在 C6 修复。

## 12. R1：Production Beta 发布

### 目标

只把 C6 已接受的候选提交发布到 Production，并完成最小、可回滚的发布验证。

### 进入条件

- C6 已完成并固定 commit SHA。
- 用户再次明确授权 Production 数据库、环境变量、Inngest 和部署写入。
- 发布窗口、迁移、备份和回滚方案已复核。

### 交付与门禁

1. 将已接受的 `main` 候选更新到 `release`，由 `release` 触发唯一 Production 部署。
2. 只应用 C1-C6 已审查的必要迁移，验证 Production 环境变量但不打印值。
3. 同步 Inngest，运行默认非付费 Production smoke。
4. 在另一次明确的付费确认和成本上限内运行一条低范围中文真实研究。
5. 发送一次不含个人信息、研究正文或 Provider payload 的受控 Sentry 异常，并确认 Analytics 只采集批准的页面访问数据。
6. 验证登录、研究、证据、报告、Settings/删除边界和 3 个公开案例。
7. 失败时停止新写入并回滚到发布前稳定 Deployment。

### 明确不做

- 不在发布过程中开发或修复非发布阻塞功能。
- 不启用 Vercel Preview。
- 不把 `main` 的后续日常提交自动带入 Production。

### 终止条件

Production smoke 和低范围真实研究通过，发布记录写入 `PROJECT_STATUS.md`；否则完成回滚并记录阻塞原因。

## 13. 路线图外候选

支付、团队、导出、定时研究、浏览器扩展、OCR 导入、自托管模型和通用聊天均不属于 C1-C6。只有 R1 完成并获得真实使用反馈后，才重新评估这些能力。
