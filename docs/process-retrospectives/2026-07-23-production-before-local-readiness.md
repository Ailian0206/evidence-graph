# 流程复盘：Production 先于本地产品完备度

日期：2026-07-23

## 1. 发生了什么

真实 Provider 代码合并后，开发连续进入 Production 数据库迁移、Vercel 环境变量、部署、Inngest 同步和真实研究验收。远端链路最终可用，但本地产品仍缺少稳定的真实运行入口、逐阶段用户验收、Settings、账号/数据删除、Evidence Eval 和 3 个真实案例。

状态文档把 PR 合并、Production 验收和产品完成放在同一个“已完成”口径里，导致用户看不到本地产品究竟完成了什么、缺什么、下一次能验收什么。

## 2. 影响

- 开发优先级从本地产品闭环偏移到发布基础设施。
- Production 被用于发现本应在本地发现的真实链路问题。
- 技术提交很多，但用户缺少连续可感知的本地产品进展。
- 当前状态页保留了大量 CI 和部署流水，却没有清楚表达 MVP 缺口。

已完成的 Provider、数据库和部署工作仍然有效，不回滚；需要修正的是后续推进顺序和完成度口径。

## 3. 根因

1. 原开发计划以技术模块和 PR 为主，没有把用户本地验收写成硬门禁。
2. “代码已合并”“Production 可运行”“MVP 产品完成”没有分开记录。
3. 每个阶段没有明确的“不做”和“终止条件”，完成一个技术边界后容易继续向部署扩张。
4. 真实 Provider 的首次完整验收选择了 Production，缺少等价的本地运行环境。
5. 状态文档偏重历史证明材料，弱化了当前阶段和下一次用户验收。

## 4. 修正后的规则

1. 同一时间只允许一个有限里程碑。
2. 每个里程碑必须有用户可见结果、自动化门禁、本地验收清单、明确不做和终止条件。
3. 实现完成后先由 Agent 启动本地完整环境并提供 URL；用户明确本地验收通过后，才能创建和合并里程碑 PR，并进入下一阶段。
4. 代码完成度、本地验收度、产品完成度和 Production 部署状态分开记录。
5. C6 Release Candidate 以前冻结 Production；只有用户明确批准发布，才更新 `release`。
6. 当前里程碑结束后必须停止，不静默开始下一里程碑。
7. 真实 Provider 每次调用继续使用专用确认门禁和成本上限，常规测试保持 fixture-only。

## 5. 新顺序

```text
规划与真实进度基线
-> 本地真实研究运行环境
-> 核心闭环和 P0/P1 收敛
-> Settings 与数据生命周期
-> Evidence Eval
-> 3 个真实案例与作品集回填
-> 本地 Release Candidate
-> 用户明确批准后的 Production Beta
```

具体边界和验收标准以 `docs/roadmap.md` 为准。
